These are notes from the Chapter 1 from the book "Software Architecture: Perspective on an Emerging Discipline", by Mary Shaw and David Garlan, Prentice Hall, 1996.
  • Software Architecture becomes most important part in Software Engineering;
    What is Software Architecure? It is not the same as design. We could say it 'high level design', but it is also still not the same. Software Architecture is components + interactions, but also design decisions, the decisions why such design gives better quality than the other;
  • It is not functionality that is important, but the quality; this is because we want to
    - to deal with complexity and ease debugging
    - to support reuse
    - ...
  • Software Engineering, in turn, is a routine (which means a repeatative process); Software Architecture is one of the artifacts produced by Software Engineering; another artifact can be the code itself; Software Engineering deals with the following things:
    - solving a customer's problem
    - with certain resource limit (e.g. the money)
    - by applying a systematic approach (repeatable process)

* the routine is for cotrolling the workers who just want to do the work and go home; having only scientists and PhDs would be ideal case but not real life;
* proper architecture gives 80% chance that the program will succeed
* folklore - when people tell their colleagues about some way of doing something, and it spreads in this informal way
* there was an attempt to introduce licenses for Software Engineers but it didn't succeed as people didn't take it seriously - no one really trusts software
* for example in NASA they have voting system, i.e. 3-5 computers run exactly the same program simultaneously
